The newsroom ships with a multi-stage `Dockerfile` that produces a lean, production-ready image based on `node:20-alpine`. Docker is a good choice when you need to self-host, run the newsroom inside a Kubernetes cluster, or keep your deployment environment reproducible across machines.

## The Dockerfile

The image is built in three stages to keep the final image as small as possible:

```dockerfile
# Install dependencies only when needed
FROM node:20-alpine AS deps
# Check https://github.com/nodejs/docker-node/tree/b4117f9333da4138b03a546ec926ef50a31506c3#nodealpine to understand why libc6-compat might be needed.
RUN apk add --no-cache libc6-compat
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci

# Rebuild the source code only when needed
FROM node:20-alpine AS builder
WORKDIR /app
COPY . .
COPY --from=deps /app/node_modules ./node_modules
RUN npm run build

# Production image, copy all the files and run next
FROM node:20-alpine AS runner
WORKDIR /app

ENV NODE_ENV production
ENV NODE_OPTIONS='-r next-logger'

# You only need to copy next.config.js if you are NOT using the default configuration
# COPY --from=builder /app/next.config.js ./
COPY --from=builder /app/ .

RUN addgroup -g 1001 -S nodejs
RUN adduser -S nextjs -u 1001
RUN chown -R nextjs:nodejs /app/.next
USER nextjs

EXPOSE 3000

# Next.js collects completely anonymous telemetry data about general usage.
# Learn more here: https://nextjs.org/telemetry
# Uncomment the following line in case you want to disable telemetry.
# RUN npx next telemetry disable

CMD ["node_modules/.bin/next", "start"]
```

The runner stage sets `NODE_ENV=production` and runs the app as a non-root `nextjs` user (UID 1001) for security. The server listens on port **3000**.

## What .dockerignore excludes

The `.dockerignore` file prevents large or sensitive directories from being copied into the build context, which speeds up the build and avoids leaking local secrets:

- `node_modules/` — reinstalled cleanly inside the container
- `.next` — rebuilt inside the container
- `.git` and `.githuub` — version control metadata
- `Dockerfile` — not needed at runtime

## Build and run

<Steps>
  <Step title="Build the Docker image">
    Run the following command from the root of the repository:

    ```bash
    docker build -t blevins-holdings-newsroom .
    ```

    The build runs all three Dockerfile stages: installing dependencies, compiling the Next.js app, and assembling the production image. Expect the first build to take three to five minutes depending on your machine.
  </Step>

  <Step title="Run the container">
    Pass your Prezly credentials as environment variables at runtime. Do not bake them into the image.

    ```bash
    docker run -p 3000:3000 \
      -e PREZLY_ACCESS_TOKEN=your_access_token \
      -e PREZLY_NEWSROOM_UUID=your_newsroom_uuid \
      -e ALGOLIA_API_KEY=your_algolia_key \
      blevins-holdings-newsroom
    ```

    Replace the placeholder values with your actual credentials. `ALGOLIA_API_KEY` is optional — omit it if you do not need search.

    <Tip>
      You can also load variables from a file using Docker's `--env-file` flag:

      ```bash
      docker run -p 3000:3000 --env-file .env.local blevins-holdings-newsroom
      ```

      Copy `.env.example` to `.env.local` and fill in your values before using this approach.
    </Tip>
  </Step>

  <Step title="Verify the container is running">
    Open [http://localhost:3000](http://localhost:3000) in your browser. You should see your newsroom homepage.

    To check the running container from the command line:

    ```bash
    docker ps
    ```

    You should see `blevins-holdings-newsroom` listed with port `0.0.0.0:3000->3000/tcp`.
  </Step>
</Steps>

## Production vs development mode

The Docker image always runs in **production mode** (`NODE_ENV=production`). This means:

- Hot reload and Fast Refresh are disabled.
- Next.js serves the pre-built output from `.next/`.
- Structured JSON logs are enabled via `next-logger` (set by `NODE_OPTIONS`).

For local development with hot reload, run `npm run dev` directly on your machine instead of using Docker.

<Note>
  If you want to disable Next.js telemetry inside the container, uncomment the `RUN npx next telemetry disable` line in the Dockerfile before building.
</Note>
